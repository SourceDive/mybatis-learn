# MyBatis 源码依赖问题分析与解决过程

## 问题回顾

在搭建 MyBatis 源码学习环境的过程中，我们遇到了一个关键问题：**测试程序依赖的是外部 MyBatis 而不是本地源码**。

## 问题分析

### 1. 错误的配置方式

#### 错误配置 1：多模块项目配置不当

```xml
<!-- 错误的配置 -->
<parent>
    <groupId>com.zero.learning</groupId>
    <artifactId>mybatis-learn-parent</artifactId>
    <version>1.0-SNAPSHOT</version>
    <relativePath>../pom.xml</relativePath>
</parent>

<dependency>
    <groupId>org.mybatis</groupId>
    <artifactId>mybatis</artifactId>
    <type>jar</type>  <!-- 这里没有指定版本 -->
</dependency>
```

**问题分析**：
- 虽然配置了多模块项目，但 Maven 仍然会尝试从中央仓库解析依赖
- 没有指定版本号，Maven 会使用 `dependencyManagement` 中的版本
- 但是 `3.5.4-my-debug-snapshot` 这个版本在中央仓库不存在
- 结果：Maven 会尝试下载一个不存在的版本，或者回退到默认版本

#### 错误配置 2：依赖解析机制误解

```xml
<!-- 父 pom.xml 中的 dependencyManagement -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.mybatis</groupId>
            <artifactId>mybatis</artifactId>
            <version>${mybatis.version}</version>  <!-- 3.5.4-my-debug-snapshot -->
        </dependency>
    </dependencies>
</dependencyManagement>
```

**问题分析**：
- `dependencyManagement` 只是管理版本，不提供实际的依赖
- 子模块仍然需要从 Maven 仓库下载依赖
- 本地源码编译的 JAR 文件没有被正确引用

### 2. Maven 依赖解析机制

#### 依赖解析顺序

1. **本地仓库** (`~/.m2/repository`)
2. **中央仓库** (Maven Central)
3. **远程仓库** (配置的私有仓库)

#### 问题根源

```bash
# 当我们运行 mvn dependency:tree 时
org.mybatis:mybatis:jar:3.5.4-my-debug-snapshot:compile
```

这个输出表明：
- Maven 认为它找到了 `3.5.4-my-debug-snapshot` 版本
- 但实际上这个版本可能来自：
  1. 本地仓库（如果之前安装过）
  2. 中央仓库（如果存在）
  3. 或者是一个错误的解析结果

### 3. 正确的解决方案

#### 方案 1：使用 system 作用域（我们采用的方案）

```xml
<dependency>
    <groupId>org.mybatis</groupId>
    <artifactId>mybatis</artifactId>
    <version>3.5.4-my-debug-snapshot</version>
    <scope>system</scope>
    <systemPath>${project.basedir}/../mybatis-3/target/mybatis-3.5.4-my-debug-snapshot.jar</systemPath>
</dependency>
```

**为什么这样有效**：
- `system` 作用域明确告诉 Maven 使用本地文件系统路径
- 不会尝试从任何 Maven 仓库下载
- 直接指向本地编译的 JAR 文件

#### 方案 2：使用 Maven 多模块项目（更优雅的方案）

```xml
<!-- 父 pom.xml -->
<modules>
    <module>mybatis-3</module>
    <module>mybatis-main-dir</module>
</modules>

<!-- 子模块 pom.xml -->
<dependency>
    <groupId>org.mybatis</groupId>
    <artifactId>mybatis</artifactId>
    <version>3.5.4-my-debug-snapshot</version>
</dependency>
```

**为什么这样有效**：
- Maven 会先编译 `mybatis-3` 模块
- 然后将编译结果安装到本地仓库
- `mybatis-main-dir` 模块从本地仓库获取依赖

## 解决过程

### 步骤 1：识别问题

```bash
# 运行依赖树分析
mvn dependency:tree

# 发现问题：依赖指向外部仓库而不是本地源码
```

### 步骤 2：分析配置

```xml
<!-- 检查 pom.xml 配置 -->
<dependency>
    <groupId>org.mybatis</groupId>
    <artifactId>mybatis</artifactId>
    <type>jar</type>  <!-- 问题：没有指定版本和作用域 -->
</dependency>
```

### 步骤 3：修复配置

```xml
<!-- 修复后的配置 -->
<dependency>
    <groupId>org.mybatis</groupId>
    <artifactId>mybatis</artifactId>
    <version>3.5.4-my-debug-snapshot</version>
    <scope>system</scope>
    <systemPath>${project.basedir}/../mybatis-3/target/mybatis-3.5.4-my-debug-snapshot.jar</systemPath>
</dependency>
```

### 步骤 4：验证修复

```bash
# 重新运行依赖树分析
mvn dependency:tree

# 确认输出：
# org.mybatis:mybatis:jar:3.5.4-my-debug-snapshot:system
```

## 经验教训

### 1. 理解 Maven 依赖解析机制

- **依赖作用域**：`compile`、`test`、`provided`、`system` 等
- **依赖来源**：本地仓库、中央仓库、远程仓库
- **版本管理**：`dependencyManagement` vs 实际依赖

### 2. 多模块项目的正确使用

- **模块顺序**：Maven 会按照依赖关系编译模块
- **依赖传递**：子模块可以依赖其他子模块
- **版本统一**：使用 `dependencyManagement` 统一管理版本

### 3. 源码学习环境的最佳实践

- **明确依赖来源**：确保使用本地源码
- **自动化构建**：创建脚本简化编译流程
- **依赖验证**：使用 `dependency:tree` 验证配置

## 总结

这个问题的根本原因是**对 Maven 依赖解析机制的误解**。我们以为配置了多模块项目就能自动使用本地源码，但实际上 Maven 仍然会尝试从仓库下载依赖。

**关键教训**：
1. **明确依赖来源**：使用 `system` 作用域明确指向本地文件
2. **理解 Maven 机制**：多模块项目不等于自动使用本地源码
3. **验证配置**：每次修改后都要验证依赖关系
4. **自动化流程**：创建脚本简化操作，减少人为错误

通过这次问题解决，我们不仅修复了配置，更重要的是深入理解了 Maven 的依赖管理机制，这对后续的源码学习和开发工作都有很大帮助。
