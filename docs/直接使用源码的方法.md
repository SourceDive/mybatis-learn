# 直接使用 MyBatis 源码的方法

## 问题

你问得很好！确实可以直接指向本地源码，而不需要先编译成 JAR 文件。让我为你展示几种直接使用源码的方法。

## 方法对比

### 方法 1：使用 JAR 文件（之前的方法）

```xml
<dependency>
    <groupId>org.mybatis</groupId>
    <artifactId>mybatis</artifactId>
    <version>3.5.4-my-debug-snapshot</version>
    <scope>system</scope>
    <systemPath>${project.basedir}/../mybatis-3/target/mybatis-3.5.4-my-debug-snapshot.jar</systemPath>
</dependency>
```

**缺点**：
- 需要先编译源码生成 JAR 文件
- 每次修改源码后都要重新编译
- 依赖管理复杂

### 方法 2：直接使用源码（推荐）

```xml
<dependency>
    <groupId>org.mybatis</groupId>
    <artifactId>mybatis</artifactId>
    <version>3.5.4-my-debug-snapshot</version>
</dependency>
```

**优点**：
- 直接使用源码，无需 JAR 文件
- 修改源码后自动重新编译
- Maven 自动处理依赖关系

## 实现原理

### Maven 多模块项目机制

当使用 Maven 多模块项目时：

1. **父 pom.xml** 定义了模块关系：
```xml
<modules>
    <module>mybatis-3</module>
    <module>mybatis-main-dir</module>
</modules>
```

2. **Maven 构建顺序**：
   - 先编译 `mybatis-3` 模块
   - 将编译结果安装到本地仓库
   - 然后编译 `mybatis-main-dir` 模块
   - `mybatis-main-dir` 从本地仓库获取 `mybatis-3` 的依赖

3. **依赖解析**：
   - Maven 会优先从本地仓库获取依赖
   - 本地仓库中的 `mybatis-3.5.4-my-debug-snapshot` 就是刚刚编译的源码版本

## 验证方法

### 1. 检查构建顺序

```bash
mvn clean compile test-compile
```

**输出显示**：
```
[INFO] Reactor Build Order:
[INFO] 
[INFO] mybatis                                                            [jar]
[INFO] MyBatis 学习项目                                                       [pom]
[INFO] mybatis-main-dir                                                   [jar]
```

### 2. 检查依赖来源

```bash
mvn dependency:tree
```

**输出显示**：
```
org.mybatis:mybatis:jar:3.5.4-my-debug-snapshot:compile
```

注意：这里是 `compile` 作用域，不是 `system` 作用域。

### 3. 验证源码修改

1. 修改 `mybatis-3/src/main/java/` 中的源码
2. 运行 `mvn clean compile test-compile`
3. 观察是否重新编译了修改的文件

## 配置要求

### 1. 父 pom.xml 配置

```xml
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 
         http://maven.apache.org/xsd/maven-4.0.0.xsd">
    <modelVersion>4.0.0</modelVersion>

    <groupId>com.zero.learning</groupId>
    <artifactId>mybatis-learn-parent</artifactId>
    <version>1.0-SNAPSHOT</version>
    <packaging>pom</packaging>

    <modules>
        <module>mybatis-3</module>
        <module>mybatis-main-dir</module>
    </modules>
    
    <dependencyManagement>
        <dependencies>
            <dependency>
                <groupId>org.mybatis</groupId>
                <artifactId>mybatis</artifactId>
                <version>3.5.4-my-debug-snapshot</version>
            </dependency>
        </dependencies>
    </dependencyManagement>
</project>
```

### 2. 子模块 pom.xml 配置

```xml
<parent>
    <groupId>com.zero.learning</groupId>
    <artifactId>mybatis-learn-parent</artifactId>
    <version>1.0-SNAPSHOT</version>
    <relativePath>../pom.xml</relativePath>
</parent>

<dependencies>
    <dependency>
        <groupId>org.mybatis</groupId>
        <artifactId>mybatis</artifactId>
        <version>3.5.4-my-debug-snapshot</version>
    </dependency>
    
    <!-- 传递依赖 -->
    <dependency>
        <groupId>ognl</groupId>
        <artifactId>ognl</artifactId>
        <version>3.2.12</version>
    </dependency>
    <dependency>
        <groupId>org.javassist</groupId>
        <artifactId>javassist</artifactId>
        <version>3.26.0-GA</version>
    </dependency>
</dependencies>
```

## 工作流程

### 1. 修改源码

```bash
# 修改 mybatis-3/src/main/java/ 中的源码
vim mybatis-3/src/main/java/org/apache/ibatis/session/DefaultSqlSession.java
```

### 2. 重新编译

```bash
# 从根目录运行，Maven 会自动处理模块依赖
mvn clean compile test-compile
```

### 3. 运行测试

```bash
cd mybatis-main-dir
mvn test
```

## 优势对比

| 特性 | JAR 文件方式 | 源码直接方式 |
|------|-------------|-------------|
| 配置复杂度 | 高 | 低 |
| 修改源码后 | 需要手动编译 | 自动重新编译 |
| 依赖管理 | 手动管理 | Maven 自动管理 |
| 构建速度 | 慢 | 快 |
| 调试便利性 | 一般 | 很好 |

## 注意事项

### 1. 版本一致性

确保所有模块使用相同的版本号：
- 父 pom.xml 中的 `mybatis.version`
- 子模块中的依赖版本

### 2. 传递依赖

虽然 Maven 会自动处理传递依赖，但有时需要显式声明：
```xml
<dependency>
    <groupId>ognl</groupId>
    <artifactId>ognl</artifactId>
    <version>3.2.12</version>
</dependency>
```

### 3. 构建顺序

Maven 会按照依赖关系自动确定构建顺序，但可以通过 `<modules>` 标签明确指定。

## 总结

**直接使用源码的方法更优雅**：

1. **配置简单**：不需要 `system` 作用域和 `systemPath`
2. **自动化**：Maven 自动处理编译和依赖
3. **实时性**：修改源码后立即生效
4. **标准化**：符合 Maven 最佳实践

**推荐使用直接源码方式**，这样你可以：
- 专注于源码学习，而不是构建配置
- 享受 Maven 的自动化优势
- 获得更好的开发体验

这就是为什么 Maven 多模块项目是处理源码依赖的最佳方式！
